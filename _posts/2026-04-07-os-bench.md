# Introduction

Ever since the release of OpenClaw, I have been trying to answer one question: What is the smallest LLM running on DGX Spark that can handle a simple cronjob. In my experience, there exists a major issue with regards to "instruction following". Models will interpret the instructions in unexpected ways and cause havoc. Additionally, models will ignore or forget about instructions as the process progresses. I am partially to be blame for providing vague instructions, I ablated this by using larger models and reading their thinking traces. Those models tend to ask for clarification or they might make assumptions and continure but their assumptions are far more realistic. 

The open source (or open weight for most) models tested ranged from from Cohere [`Command A`](https://huggingface.co/unsloth/c4ai-command-a-03-2025-GGUF), [`Hunyuan`](https://huggingface.co/unsloth/Hunyuan-A13B-Instruct-GGUF) to more mainstream [`MiniMax`](https://huggingface.co/unsloth/MiniMax-M2.5-GGUF), `GLM` ([4.7 Flash](https://huggingface.co/unsloth/GLM-4.7-Flash-GGUF), [4.7 Flash Reap](https://huggingface.co/unsloth/GLM-4.7-Flash-REAP-23B-A3B-GGUF)), and `Qwen`. That is why the release of `Gemma 4` models has been a great surprise to me since I had given up on models below the ~20B size on executing tasks. `Gemma 4` series are capable way above their size group where the models below ~10B are capable of cohesive  instruction following and doing interleaved tool calling with reasoning very effectively. This inspired me to benchmark the (quantized version of) recent releases against NextJS benchmark, and also compare the relative behavior evolution of the LLMs within their family groups.

# Context

The original release of [Next Js benchmark](https://github.com/vercel/next-evals-oss) is interesting since the tasks would manually pass files within the prompts and give a specific ask then the model had to return the changes to files without tool calls or anything. This strictly a coding-related benchmark but I thought it may be a good proxy with problems I am facing in harnesses. The LLMs are trained to do tool calls and handle these kind of tasks but the prompts strictly say what needs to be done. Note that the prompt does not sat <B>NO TOOLS</B> but the model is expected to pick it up and interpret it.


Here is the specific format:

````
```
Please provide your changes as complete file contents. Use this format for each file you need to modify, create, or delete:
```file=path\/to\/file.tsx action=replace
complete file content here\n```

For example:
```file=app\/Component.tsx action=replace\nexport default function Component() {\n const new = \"code\";\n return <div>{{new}}<\/div>;
}\n```

```file=app\/NewComponent.tsx action=add
export default function NewComponent() {\n return <div>New component<\/div>;
}\n```

```file=app\/OldComponent.tsx action=delete```

Rules:
	- Always specify an action: add, replace, or delete
	- For add\/replace: provide the complete file content, not partial changes
	- For delete: you can leave the content empty or omit it
	- Use the exact file path relative to the project root
	- Maintain proper indentation and formatting
	- Include all imports and exports that the file needs
	- Do not modify *.test.tsx files
```
````

I personally do not get anything from the model cards that are released everyday with each new model release (I guess that is why there is benchmarking mini-industry being formed) But with the large volume and fast pace release of open source models, I wanted to assess progress in the simplest way possible. That is benchmarking models on NextJS benchmark to track tool calls, reasoning efficiency (or output generation length) and speed. 


# Progress

Qwen3.5 was released on <b><i>February 16, 2026</i></b>, and Gemma 4 on <b><i>April 2, 2026</i></b>. Both releases are natively multimodal with heavy emphasis on reasoning and agentic capability use. Each release has a number of smaller and mid-size models; the larger models are either MoE and Dense. Gemma uses the Per-Layer Embeddings approach for smaller releases, where the effective computation amount is lower than actual model size (E2B with 5B LLM, E4B with 9B LLM). It is interesting to observe the effect of these architectural differences.


Graph below compares NextJS pass@1 for Qwen3.5, Gemma4, Nemotron, and GLM-4.7 Flash. I define **Reasoning efficiency** as the squared reward divided by log10 of reasoning length. The metric aims to ensure the amount of tokens generated for reasoning is worth it. Easiest way to think about it: smaller Qwen models (< 8B) generate long reasoning traces riddled with ***"oh wait"s*** with not a worthwhile response (not all tasks just complicated ones). The other metric that is important is tool call percentage to check what portion of responses involved doing tool calls.


Figure below shows these metrics across family models. Let's go through each family then we can compare similar sizes across families:
- The best models here are with Gemma 4, that is the case for both small and larger ones.
- Gemma 4 series is exceptional at understanding that no tool calls are required. Specially the smaller ones, Gemma4-E2B and Gemma4-E4B are on par with Qwen3.5-9B except for slight drop in reasoning efficiency. 
- Qwen3.5 model below 9B size, are doing extremenly bad with almost all the responses being tool calls. 
- Nemotron Cascade-2 is the post-tuned version of ``Nano-30B-A3B``, we can see the difference: 10% reward increase, no tool calls in cascade-2 version, and it has a stronger reasoning efficiency. The [Cascade-2 paper](https://arxiv.org/abs/2603.19220) stresses the order of training data used to ensure different capabilities (reasoning on math vs coding vs instruction following) are not forgotten/overridden in between runs. And the results show significant improvements across the metrics.
- GLM-4.7-flash underperforms compared to Gemma4-26B-A4B but does better than Qwen3.5-35B-A3B. The REAP-ed version of it does worse as well where every metric gets worse.
- The bigger models have done relatively worse specially nemotron super 120B but presumably post-tuning in cascade-2 fashion could improve it.

# Unnecessary Commands

This benchmark does not require tool calls but the prompts is being editing code which fundamentally relies on harnesses functionalities for editing. So it makes it a prime candidate to evaluate if the models refrain from tool calls and stick to the instruction.

Graph below shows the CMD distrubitions for all the runs.

A more detailed view shows that release families do have a different distributions of tool calls, some observations:
- Gemma 4 has the most tame cmds where there are just for EDA. And also Gemma 4 has an amazing capacity to follow through the instructions.
- The Qwen3 series and subseries (Qwen3-VL, Qwen3.5) all show a diverse range of CMDs pinpointing a very bad understanding of how to handle the issue. The main uptick is within the smaller models making them fragile to handle tool calling and reasoning.
- Gemma 4 Instruct version does worse than Gemma 4 thinking but still way better than Qwen releases.
